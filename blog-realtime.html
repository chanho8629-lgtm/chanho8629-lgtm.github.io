Overview

WebSocket + STOMP + RabbitMQ로 실시간 시스템 구축

BIDEO 경매 플랫폼에서 입찰이 발생하면 수백 명의 참여자에게 동시에 실시간으로 정보가 전달되어야 합니다. 단순한 폴링(polling)은 서버 부하와 지연 문제가 있어, WebSocket 기반 양방향 통신과 STOMP 프로토콜을 선택했습니다. 이벤트 기반 비동기 처리는 RabbitMQ로, 다중 서버 환경의 메시지 브로드캐스트는 Redis Pub/Sub으로 처리했습니다. AI 응답 스트리밍처럼 단방향 서버→클라이언트 스트리밍이 필요한 경우 SSE(Server-Sent Events)를 활용했습니다.

클라이언트 ⇄ WebSocket / STOMP → Spring MessageBroker → RabbitMQ → Redis Pub/Sub → 전체 서버 브로드캐스트
Core Topics

핵심 학습 주제

🔌
WebSocket — 양방향 지속 연결
HTTP는 요청-응답이 끝나면 연결이 끊어지지만, WebSocket은 TCP 연결을 유지하여 서버가 먼저 데이터를 보낼 수 있습니다. Upgrade 핸드셰이크 과정을 이해하고, 연결 끊김 감지(Heartbeat/Ping-Pong)와 자동 재연결(Exponential Backoff) 전략을 구현했습니다. 연결당 세션 객체를 관리하는 방식으로 특정 사용자에게 개인화 메시지를 전송했습니다.
📨
STOMP 프로토콜 — WebSocket 위의 메시징 레이어
WebSocket은 바이트 스트림만 다루지만, STOMP는 그 위에 pub/sub 패턴과 헤더를 제공합니다. /topic/ (브로드캐스트)와 /queue/ (개인 메시지) 두 채널로 구독을 분리했습니다. @MessageMapping으로 메시지를 수신하고 @SendTo로 특정 토픽에 발행하는 구조가 직관적이고 테스트하기 쉬웠습니다.
🐇
RabbitMQ — Exchange/Queue/Binding 구조
Direct Exchange(입찰 처리), Fanout Exchange(전체 알림), Topic Exchange(카테고리별 구독) 세 유형을 실제로 적용해봤습니다. Dead Letter Queue로 처리 실패 메시지를 격리하고, 메시지 TTL과 최대 재시도 횟수를 설정하여 무한 재시도를 방지했습니다. Spring AMQP와 RabbitAdmin으로 Queue/Exchange를 코드에서 선언했습니다.
📡
SSE — 단방향 서버→클라이언트 스트리밍
WebSocket 연결이 부담스러운 경우(알림, AI 응답 스트리밍)에 SSE를 활용했습니다. SseEmitter로 서버에서 이벤트를 밀어 넣고, EventSource API로 클라이언트가 수신했습니다. 연결 만료(timeout) 처리와 클라이언트 재연결 시 Last-Event-ID로 이전 이벤트를 재전송하는 메커니즘을 구현했습니다.
🔴
Redis Pub/Sub — 다중 서버 브로드캐스트
서버가 2대 이상이면 WebSocket 세션이 서버마다 분리됩니다. Redis Pub/Sub으로 한 서버가 발행한 메시지를 모든 서버가 수신해 자신의 WebSocket 세션에 브로드캐스트하는 구조를 구현했습니다. Redisson의 RTopic을 사용하여 구독 관리를 단순화했습니다.
Code Concepts

핵심 코드 개념

WebSocket STOMP 설정 — WebSocketConfig.java
// WebSocket STOMP 설정 @Configuration @EnableWebSocketMessageBroker public class WebSocketConfig implements WebSocketMessageBrokerConfigurer { @Override public void configureMessageBroker(MessageBrokerRegistry config) { // 브로드캐스트 채널 및 개인 메시지 채널 설정 config.enableSimpleBroker("/topic", "/queue"); config.setApplicationDestinationPrefixes("/app"); config.setUserDestinationPrefix("/user"); } @Override public void registerStompEndpoints(StompEndpointRegistry registry) { registry.addEndpoint("/ws") .setAllowedOriginPatterns("*") .withSockJS(); // SockJS 폴백 지원 } }
입찰 메시지 처리 — BidController.java
// 입찰 메시지 수신 및 브로드캐스트 @Controller @RequiredArgsConstructor public class BidController { private final BidService bidService; private final SimpMessagingTemplate messagingTemplate; @MessageMapping("/bid/{auctionId}") public void processBid( @DestinationVariable Long auctionId, BidRequest bid, Principal principal ) { BidResponse result = bidService.placeBid(auctionId, bid, principal.getName()); // 경매방 전체에 새 입찰 정보 브로드캐스트 messagingTemplate.convertAndSend( "/topic/auction/" + auctionId, result ); // 이전 최고가 입찰자에게 개인 알림 messagingTemplate.convertAndSendToUser( result.getOutbidUserId(), "/queue/notifications", new OutbidNotification(auctionId, result.getNewPrice()) ); } }
Redis Pub/Sub 브로드캐스트 — RedisMessageBroker.java
// 다중 서버 환경 Redis Pub/Sub 브로드캐스트 @Component @RequiredArgsConstructor public class RedisMessageBroker { private final RedissonClient redisson; private final SimpMessagingTemplate messagingTemplate; @PostConstruct public void subscribe() { RTopic topic = redisson.getTopic("auction.bid.events"); topic.addListener(BidEvent.class, (channel, event) -> { // 이 서버의 WebSocket 세션에 브로드캐스트 messagingTemplate.convertAndSend( "/topic/auction/" + event.getAuctionId(), event ); }); } public void publish(BidEvent event) { redisson.getTopic("auction.bid.events").publish(event); } }
Case Study

BIDEO 경매 시스템 — 실시간 입찰 구현

⚡ 동시 입찰 처리

여러 사용자가 동시에 입찰하면 경쟁 조건(Race Condition)이 발생합니다. Redis 분산 락(Redisson RLock)으로 경매당 하나의 입찰만 처리되도록 보장했습니다. 락 획득 대기 시간 500ms, 만료 시간 3초로 설정했습니다.

📡 낙찰 알림 및 정산

경매 종료 시 WebSocket으로 낙찰자에게 즉시 알림을 전송하고, RabbitMQ로 정산·이메일 발송·영상 소유권 이전 작업을 비동기로 처리했습니다. 정산 실패 시 Dead Letter Queue에서 수동 처리합니다.

🔄 연결 끊김 복구

클라이언트 네트워크가 불안정할 경우 SockJS가 Long Polling으로 자동 전환됩니다. 재연결 시 최신 경매 상태를 REST API로 동기화하고, 그 이후부터 WebSocket 이벤트를 다시 수신하는 전략을 구현했습니다.

🔐 WebSocket 인증

HTTP 핸드셰이크 단계에서 JWT를 검증하는 HandshakeInterceptor를 구현했습니다. STOMP CONNECT 프레임의 Authorization 헤더를 파싱하여 Principal을 설정하고, 인증되지 않은 연결은 즉시 차단했습니다.

Lessons Learned

개발하며 배운 것들

  • WebSocket 세션 관리 — 세션이 메모리에 쌓이면 서버가 죽습니다. 연결 수를 모니터링하고, 비활성 세션을 주기적으로 정리하는 로직이 필수입니다.
  • 동시성 처리 — synchronized vs Redis 락 — synchronized는 단일 서버에서만 동작합니다. 스케일 아웃을 고려한다면 처음부터 Redis 분산 락을 사용하는 것이 올바른 선택입니다.
  • 메시지 유실 방지 — RabbitMQ 메시지를 처리하다 서버가 죽으면 ack가 없으므로 메시지가 재발행됩니다. 멱등성(Idempotency)을 보장하도록 중복 처리 방지 로직이 반드시 필요합니다.
  • SSE vs WebSocket 선택 기준 — 단방향 스트리밍은 SSE가 구현이 단순하고 HTTP/2와 잘 맞습니다. 양방향 소통이 필요하거나 낮은 지연이 중요하면 WebSocket을 선택해야 합니다.
