Overview

PostgreSQL · Redis · MySQL로 데이터 계층 설계

BIDEO와 Clausion 두 프로젝트에서 서로 다른 DB 스택을 선택하고 그 이유를 직접 검증했습니다. BIDEO는 복잡한 조인과 JSONB 타입이 필요하여 PostgreSQL을, Clausion은 기존 팀 숙련도를 고려하여 MySQL을 선택했습니다. 두 프로젝트 모두 Redis를 캐싱 및 세션 저장소로, Flyway를 DB 마이그레이션 도구로 활용했습니다. EXPLAIN ANALYZE로 쿼리 실행 계획을 분석하고, 복합 인덱스를 설계하여 핵심 쿼리의 응답 시간을 80% 단축했습니다.

Core Topics

핵심 학습 주제

🐘
PostgreSQL — ACID 트랜잭션 & JSONB
입찰 데이터는 ACID 트랜잭션이 절대적으로 중요합니다. PostgreSQL의 SERIALIZABLE 격리 수준으로 동시 입찰 간 데이터 정합성을 보장했습니다. 영상 메타데이터처럼 구조가 유동적인 데이터는 JSONB 컬럼에 저장하고, GIN 인덱스로 JSONB 내부 필드 검색을 최적화했습니다.
🔴
Redis — 캐싱 전략과 Pub/Sub
Cache-Aside 패턴으로 경매 현황(TTL 30초)과 사용자 프로필(TTL 10분)을 캐싱했습니다. Write-Through 패턴은 캐시와 DB를 동시에 갱신하여 정합성을 보장하지만 쓰기 성능 부담이 있어, 읽기가 압도적으로 많은 데이터에만 적용했습니다. Redis Sorted Set으로 실시간 입찰 순위를 O(log N)에 관리했습니다.
🔄
Flyway — SQL 마이그레이션 버전 관리
팀원 3명이 동시에 스키마를 변경하면 충돌이 발생합니다. Flyway의 파일명 규칙(V{version}__{description}.sql)으로 실행 순서를 보장하고, 체크섬으로 기존 마이그레이션 파일 변경을 감지합니다. 기능 브랜치마다 마이그레이션 파일 번호를 예약하는 팀 컨벤션을 수립하여 충돌을 방지했습니다.
📑
인덱스 최적화 — EXPLAIN ANALYZE
Seq Scan이 발생하는 쿼리를 EXPLAIN ANALYZE로 찾아 인덱스를 추가했습니다. WHERE 절과 ORDER BY 절을 함께 커버하는 복합 인덱스로 Extra Sort 단계를 제거했습니다. 인덱스를 과다 설정하면 INSERT/UPDATE 성능이 저하되므로, 실제 쿼리 패턴을 분석하여 꼭 필요한 것만 생성했습니다.
🐳
Docker Compose — 로컬 개발 환경 컨테이너화
팀원마다 DB 설정이 다르면 "내 컴퓨터에서는 됩니다" 문제가 생깁니다. Docker Compose로 PostgreSQL, Redis, RabbitMQ를 정의하여 docker compose up 명령 하나로 동일한 환경을 구성했습니다. Named Volume으로 데이터를 영속화하고, health check로 서비스 준비 순서를 제어했습니다.

핵심 코드 개념

복합 인덱스 & 실행 계획 분석 — PostgreSQL
-- 경매 상태별 검색에 복합 인덱스 적용 CREATE INDEX idx_auction_status_created ON auctions(status, created_at DESC) WHERE status = 'ACTIVE'; -- 부분 인덱스로 크기 최소화 -- 입찰 테이블 복합 인덱스 (경매 + 금액 정렬) CREATE INDEX idx_bids_auction_amount ON bids(auction_id, bid_amount DESC); -- EXPLAIN ANALYZE로 실행 계획 확인 EXPLAIN ANALYZE SELECT v.*, u.nickname, COUNT(b.id) AS bid_count FROM videos v JOIN users u ON v.creator_id = u.id LEFT JOIN bids b ON b.video_id = v.id WHERE v.category = 'TUTORIAL' AND v.created_at >= NOW() - INTERVAL '30 days' GROUP BY v.id, u.nickname ORDER BY bid_count DESC LIMIT 20; -- JSONB 내부 필드 GIN 인덱스 CREATE INDEX idx_video_metadata_gin ON videos USING GIN (metadata);

Flyway 마이그레이션 — V2 예시
-- V2__add_auction_tables.sql CREATE TABLE auctions ( id BIGSERIAL PRIMARY KEY, video_id BIGINT NOT NULL REFERENCES videos(id), seller_id BIGINT NOT NULL REFERENCES users(id), start_price NUMERIC(12,2) NOT NULL, status VARCHAR(20) NOT NULL DEFAULT 'SCHEDULED', starts_at TIMESTAMPTZ NOT NULL, ends_at TIMESTAMPTZ NOT NULL, metadata JSONB, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE bids ( id BIGSERIAL PRIMARY KEY, auction_id BIGINT NOT NULL REFERENCES auctions(id), bidder_id BIGINT NOT NULL REFERENCES users(id), bid_amount NUMERIC(12,2) NOT NULL, placed_at TIMESTAMPTZ DEFAULT NOW(), CONSTRAINT uq_bid_unique UNIQUE (auction_id, bidder_id, bid_amount) );

Redis 캐싱 전략

데이터 전략 키 패턴 TTL 이유
경매 현황 Cache-Aside auction:{id}:status 30초 입찰 빈도 높음, 약간의 지연 허용

Lessons Learned

개발하며 배운 것들

  • N+1 문제 해결 — JPA에서 연관 엔티티를 반복 조회하면 쿼리가 N+1개 발생합니다. @EntityGraph로 한 번에 Fetch Join하거나, @BatchSize로 IN 쿼리로 묶어 해결했습니다. Hibernate 통계 모드로 실제 쿼리 수를 모니터링했습니다.
  • 트랜잭션 격리 수준 — READ COMMITTED(기본값)는 Phantom Read가 발생합니다. 경매 입찰처럼 정합성이 중요한 로직은 REPEATABLE READ로 설정했습니다. 격리 수준이 높을수록 성능 비용이 있으므로 도메인별로 적절히 선택했습니다.
  • 인덱스 과다 설정 문제 — 인덱스가 많을수록 INSERT/UPDATE/DELETE 시 인덱스도 함께 갱신됩니다. 실제 쿼리 패턴을 slow query log로 분석하고, 사용되지 않는 인덱스는 과감히 제거했습니다.
  • Redis Cache Stampede — TTL이 만료되면 동시에 많은 요청이 DB에 몰립니다. Mutex Lock(SETNX)으로 하나의 요청만 DB를 조회하고 나머지는 대기하는 패턴으로 해결했습니다.
  • Flyway 팀 컨벤션 — 마이그레이션 파일 번호 충돌을 막기 위해 기능 브랜치마다 번호 대역을 사전 배정(예: feature-A는 V100~, feature-B는 V200~)하는 규칙을 도입했습니다.
